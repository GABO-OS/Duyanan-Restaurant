================================================================================
MODULE: ORDER TRACKING & FULFILLMENT DASHBOARD
================================================================================

Module Description:
  This module covers the "Orders" tab inside the Admin Panel. It allows the
  Admin/Kitchen Staff to:
    - View all incoming customer orders in a table (newest first).
    - Update an order's fulfillment status using a dropdown in the Action column.
    - Cancel an order by selecting "Cancelled" from the dropdown, which triggers
      a mandatory cancellation reason popup (SweetAlert2 modal).

================================================================================
USER STORIES
================================================================================

US-01 | View Incoming Orders
  As an Admin / Kitchen Staff,
  I want to view all incoming customer orders in a table sorted from newest to
  oldest with full order details visible at a glance,
  So that I can efficiently monitor and manage all active orders without missing
  any new customer requests.

US-02 | Update Order Fulfillment Status
  As an Admin / Kitchen Staff,
  I want to update the fulfillment status of any order (e.g., from Pending to
  Preparing or Completed) using a simple dropdown in the Orders table,
  So that the order progress is accurately tracked and reflected in real time
  across the admin dashboard.

US-03 | Cancel an Order with a Mandatory Reason
  As an Admin,
  I want to cancel an order only after providing a mandatory cancellation reason
  through a confirmation popup,
  So that every cancellation is documented with a justification, ensuring
  transparency and data integrity for the restaurant records.

================================================================================
FEATURE 1 - VIEW INCOMING ORDERS DASHBOARD
================================================================================

Requirement ID  : REQ-FBD-030 to REQ-FBD-032
Description     : The Orders tab must display all customer orders sorted from
                  newest to oldest. Staff must be able to see all order details
                  at a glance. The system must gracefully handle an empty state
                  when there are no orders.

Acceptance Criteria:
  GWT-01:
    Given  the Admin is logged in and navigates to the Orders tab,
    When   the Orders tab loads,
    Then   a table must be displayed with the columns:
           Order ID | Customer Info | Items | Timestamp | Price | Status | Action,
           and three stat boxes (Pending, Preparing, Completed) must appear at
           the top showing the correct count for each status.

  GWT-02:
    Given  at least two orders exist with different timestamps,
    When   the Admin views the Orders table,
    Then   the orders must be sorted in descending order by timestamp
           (newest order appears at the top of the list).

  GWT-03:
    Given  no orders exist in the database,
    When   the Admin navigates to the Orders tab,
    Then   the table body must display the message "No active orders.",
           all three stat boxes must show 0, and no errors or crashes occur.

  GWT-04:
    Given  the Admin navigates to the Orders tab,
    When   the page sends a request to GET /api/admin/orders,
    Then   the API must return HTTP 200 OK with the order data successfully
           rendered in the table.

Preconditions:
  - Admin is logged in with a valid admin account.
  - The Admin Panel is accessible at http://localhost:5173.
  - At least two orders exist in the database with different timestamps (for
    TC-ORD-001 and TC-ORD-002).
  - For TC-ORD-003: The orders table in the test database has been cleared.

FEATURE 2 - UPDATE ORDER FULFILLMENT STATUS
================================================================================

Requirement ID  : REQ-FBD-040 to REQ-FBD-042
Description     : Admin/Kitchen staff must be able to update an order's status
                  to a valid state (PREPARING, COMPLETED) using the dropdown in
                  the Action column. The system must reject updates to
                  non-existent orders and must not wipe existing order data.

Acceptance Criteria:
  GWT-01:
    Given  the Admin is on the Orders tab and an existing order is visible,
    When   the Admin selects a new valid status (e.g., "Completed" or
           "Preparing") from the Action column dropdown,
    Then   the Status badge in that order's row must update to the new status
           immediately, and a SweetAlert2 toast with title "Status Updated"
           must appear and auto-dismiss.

  GWT-02:
    Given  the Admin is on the Orders tab,
    When   a PUT request is made to /api/admin/orders/999999/status for an
           order ID that does not exist in the database,
    Then   the API must return HTTP 404 Not Found, and the UI must display a
           SweetAlert2 error notification.

  GWT-03:
    Given  an existing order (e.g., Order #15) with 2 items and a total of
           P500 is visible in the Orders table,
    When   the Admin updates the order's status using the Action dropdown,
    Then   the Items column and the Price column for that order must remain
           unchanged — only the Status badge must reflect the new status.

Preconditions:
  - Admin is logged in.
  - For TC-STS-001 and TC-STS-003: A PENDING order exists (e.g., Order #15)
    with 2 items and a total of P500.
  - For TC-STS-002: Order ID 999999 does NOT exist in the database.

FEATURE 3 - ORDER CANCELLATION & REASON VALIDATION
================================================================================

Requirement ID  : REQ-FBD-050 to REQ-FBD-052
Description     : When Admin selects "Cancelled" from the Action dropdown, a
                  mandatory SweetAlert2 popup must appear requiring a
                  cancellation reason. The system must block cancellation if the
                  reason field is blank or contains only whitespace.

Acceptance Criteria:
  GWT-01:
    Given  the Admin is on the Orders tab with an existing PENDING order,
    When   the Admin selects "Cancelled" from the Action column dropdown,
    Then   a SweetAlert2 modal with the title "Cancellation Reason Required"
           must appear, containing a mandatory textarea and the buttons
           "Confirm Cancellation" (red) and "Go Back" (gray).

  GWT-02:
    Given  the Cancellation Reason modal is open,
    When   the Admin types a valid, non-empty reason and clicks
           "Confirm Cancellation",
    Then   the modal must close, an "Order Cancelled" success toast must appear,
           the Status badge must update to a red CANCELLED badge, and the
           cancellation reason must be displayed below the badge and saved
           in the database.

  GWT-03:
    Given  the Cancellation Reason modal is open,
    When   the Admin clicks "Confirm Cancellation" while the textarea is empty,
    Then   the modal must remain open and display the validation error:
           "You must provide a cancellation reason!"
           and no cancellation API request must be sent.

  GWT-04:
    Given  the Cancellation Reason modal is open,
    When   the Admin enters only whitespace (spaces) in the textarea and clicks
           "Confirm Cancellation",
    Then   the modal must remain open and display the same validation error:
           "You must provide a cancellation reason!"
           and no cancellation API request must be sent.

  GWT-05:
    Given  the Cancellation Reason modal is open,
    When   the Admin clicks "Go Back" without entering a reason,
    Then   the modal must close, the order's status must remain unchanged,
           and the Action dropdown must revert to the order's original value.

Preconditions:
  - Admin is logged in and on the Orders tab.
  - An existing PENDING order is available (e.g., Order #21).

TC-CAN-003 | REQ-FBD-052 | Cancellation Blocked - Whitespace-Only Reason
--------------------------------------------------------------------------------
Objective   : Validate that entering only spaces (whitespace) in the
              cancellation reason textarea is correctly rejected by the frontend.
Test Scenario: Validate that a whitespace-only cancellation reason is
               rejected identically to an empty input.

Test Steps  :
  1. Log in as Admin and navigate to the Orders tab.
  2. Locate any PENDING order and select "Cancelled" from the Action dropdown.
  3. The SweetAlert2 "Cancellation Reason Required" modal appears.
  4. In the textarea, type ONLY spaces (e.g., press the spacebar 5+ times).
     The input appears to have content visually, but is all whitespace.
  5. Click "Confirm Cancellation".
  6. Observe the result.

Expected Result:
  - The popup does NOT close.
  - The same validation error message appears:
      "You must provide a cancellation reason!"
    (The SweetAlert2 inputValidator checks: !value.trim() — meaning a
     whitespace-only string is treated the same as an empty string.)
  - The order status remains unchanged.
  - No cancellation API request is sent.

Frontend Note: This behavior is enforced by the following code in AdminPanel.jsx:
  inputValidator: (value) => {
    if (!value || !value.trim()) {
      return 'You must provide a cancellation reason!';
    }
  }

Priority    : High
Test Data   : See TEST DATA - FEATURE 3 section below.
